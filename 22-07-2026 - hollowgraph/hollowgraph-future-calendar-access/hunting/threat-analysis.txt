FAR-FUTURE MICROSOFT GRAPH CALENDAR ACCESS
Classification: Hunting query
Confidence: High for observability; Medium for maliciousness
Research date: 2026-07-22

1. Objective

Identify Microsoft Graph calendarView requests that target years 2040 through
2099. The query generalizes the far-future calendar access observed in
HOLLOWGRAPH while preserving a simple, reviewable condition.

2. Source-observed fact

Group-IB observed HOLLOWGRAPH querying a calendar view for 2050-05-13 to
retrieve tasking from a calendar event unlikely to be noticed by the mailbox
owner.

3. Analyst interpretation

A calendarView request many years in the future is unusual and may expose
similar dead-drop behavior even if an attacker changes the exact date.
However, the source does not establish that all far-future calendar requests
are malicious, nor does it justify a universal alert threshold. The logic is
therefore hunting-only until tenant history is measured.

4. Required telemetry

Native Log Analytics table:
- MicrosoftGraphActivityLogs

Required fields:
- TimeGenerated
- RequestMethod
- RequestUri
- ResponseStatusCode
- AppId
- ServicePrincipalId
- UserId
- IPAddress
- UserAgent

Known schema and formatting assumption:
- RequestUri retains a readable startDateTime query parameter in the form
  startDateTime=20YY-. URL transformation or truncation may prevent matching.

5. Why the conditions exist

- GET: focuses on calendar retrieval rather than event creation.
- /calendarView: matches the endpoint observed in the research.
- startDateTime years 2040-2099: captures far-future access without claiming
  that it is inherently malicious.
- No success-code filter: hunting retains failed probes for analyst context.
- No aggregation or threshold: neither is justified by the source evidence.

6. Expected false positives

- application and integration testing;
- migration or archival tooling;
- long-range planning and scheduling products;
- vulnerability or quality-assurance testing;
- malformed requests generated by internal software.

7. Validation experiment

1. Run the query over 30 days, then extend to the available retention period.
2. Group results manually by AppId, ServicePrincipalId, UserId, and UserAgent.
3. Attribute each application to an owner and documented business purpose.
4. Test different URI encodings to determine whether the regex misses encoded
   or transformed parameters.
5. Review Graph permissions and sign-in context for unidentified applications.

8. Promotion criteria

A production rule may be considered only if:

- RequestUri consistently exposes the required parameter;
- historical legitimate volume is low and explainable;
- approved applications can be excluded using stable identities;
- test traffic is detected consistently;
- alert enrichment includes application, service principal, user, IP address,
  user agent, and response status.

9. Failure conditions

Do not promote the query when RequestUri does not expose startDateTime, when
normal applications generate unresolved recurring matches, or when identity
context is insufficient to distinguish approved applications.

10. ATT&CK mapping

- T1102.002 - Web Service: Bidirectional Communication
- T1071.001 - Application Layer Protocol: Web Protocols
